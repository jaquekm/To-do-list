1. Visão de produto

Problema resolvido e proposta de valor

Dor do mercado

  • Pequenas e médias empresas não completam um container sozinhas.
  • O rateio costuma ser manual, opaco e sujeito a conflito comercial.
  • Há baixa previsibilidade de custo antes do fechamento do embarque.
  • Coordenação documental entre múltiplos participantes é fragmentada.

Resultado esperado

  • Reservas digitais de espaço em containers compartilhados.
  • Cálculo automático de ocupação, capacidade residual e custo individual.
  • Simulador de economia vs. embarque dedicado por empresa.
  • Base para evolução em marketplace logístico B2B com matching inteligente.

Arquitetura modular monolítica no MVP, evoluindo para microserviços apenas nos domínios com forte escala/isolamento.

  • Módulo Cadastro: empresas, usuários, produtos, origens, destinos, templates documentais.
  • Módulo Embarques: criação, reserva, ocupação, consolidação, fechamento e milestones.
  • Módulo Rateio: regras, pesos de modelo híbrido, exceções e memória de cálculo.
  • Módulo Financeiro: custos, margem, contas a receber, faturamento por participante.
  • Módulo Documentos: invoice, packing list, BL, checklists e bloqueios.
  • Módulo Analytics: ocupação, rentabilidade, recorrência e forecast.

Frontend recomendado

  • Workspace do operador: visão Kanban/lista de embarques, capacidade, rentabilidade e riscos.
  • Workspace do cliente: cotação, reserva, upload documental, previsão de custo e rastreio.
  • Camada de design system: componentes padronizados para formulários complexos, tabelas auditáveis e dashboards.
  • Explicabilidade: toda tela de custo deve expor fórmula, base de rateio, percentuais e exceções aplicadas.

Regras de modelagem importantes

  • Multi-tenant: separar dados por operador logístico/holding ou usar schema por tenant em contas enterprise.
  • Versionamento: toda regra de rateio e resultado precisa de versão para reprocessamento histórico.
  • Auditoria first-class: cálculo não deve ser apenas derivado em tela; precisa ser persistido por item de custo.
  • Documentos desacoplados: armazenar metadados no banco e binários em object storage com hash.
4. Algoritmo de rateio

Lógica detalhada, auditável e configurável

Premissas de cálculo

  1. Somar peso, volume, quantidade de volumes e valor FOB/CIF por participante.
  2. Identificar o fator limitante do container: ocupação por peso ou por volume.
  3. Aplicar faixas de preço conforme ocupação total do embarque.
  4. Ratear cada custo segundo sua própria base configurada.
  5. Aplicar mínimos, sobretaxas individualizadas e arredondamento financeiro.
  6. Persistir memória de cálculo por empresa, por item de custo e por regra.

Tipos de custos suportados

  • Compartilhados: frete internacional, seguro global, THC, capatazia, armazenagem comum.
  • Individualizados: carga perigosa, inspeção específica, embalagem especial, licenças.
  • Condicionais: custo adicional por faixa de ocupação, urgência, porto de origem ou destino.
  • Margem: markup fixo, percentual ou híbrido por empresa/embarque.

Pseudocódigo do motor de rateio

function calcularRateio(embarque, reservas, custos, perfil) {
  validarPendenciasDocumentais(embarque, reservas)

  totais = agregarReservas(reservas)
  ocupacaoPeso = totais.pesoKg / embarque.container.maxWeightKg
  ocupacaoVolume = totais.volumeM3 / embarque.container.maxVolumeM3
  fatorLimitante = ocupacaoPeso >= ocupacaoVolume ? "PESO" : "VOLUME"

  faixa = descobrirFaixaDePreco(perfil.faixas, max(ocupacaoPeso, ocupacaoVolume))
  custosAjustados = aplicarFaixaNosCustos(custos, faixa)

  resultado = []

  for custo in custosAjustados {
    if (!custo.isShared) {
      reserva = localizarReserva(custo.bookingId)
      resultado.push(lancarCustoIndividual(reserva, custo))
      continue
    }

    regra = buscarRegra(perfil, custo.costType)
    baseTotal = 0
    basesPorReserva = {}

    for reserva in reservas {
      base = calcularBase(regra, reserva, fatorLimitante)
      basesPorReserva[reserva.id] = base
      baseTotal += base
    }

    for reserva in reservas {
      proporcao = baseTotal === 0 ? 0 : basesPorReserva[reserva.id] / baseTotal
      valorRateado = custo.amount * proporcao
      valorRateado += aplicarSobretaxasEspecificas(reserva, custo)
      valorRateado = aplicarMinimoEmpresa(valorRateado, regra.minimumAmount, perfil.minCharge)

      resultado.push({
        bookingId: reserva.id,
        costItemId: custo.id,
        basis: regra.basis,
        basisValue: basesPorReserva[reserva.id],
        ratio: proporcao,
        allocatedAmount: arredondar(valorRateado, 2)
      })
    }
  }

  return consolidarPorEmpresa(resultado)
}

Modelo híbrido

Permite compor a base de rateio com pesos percentuais, por exemplo: 50% volume + 30% peso + 20% valor da mercadoria.

Fórmula: base_híbrida = (volume_norm × 0,5) + (peso_norm × 0,3) + (valor_norm × 0,2)

Fator limitante

Se o container “cubou” antes de atingir o peso máximo, custos de frete podem privilegiar volume; se estourou peso antes, priorizar kg evita subsídio cruzado.

Transparência

Cada resultado deve mostrar: custo original, regra aplicada, base de cálculo, proporção, sobretaxa, mínimo e valor final faturado.

8. Stack tecnológica

Tecnologias reais e justificadas

Stack sugerida

  • Frontend: React + TypeScript + Next.js + TanStack Query + Chart.js/ECharts
  • Backend: Node.js (NestJS) + TypeScript
  • Banco transacional: PostgreSQL
  • Cache/fila: Redis + BullMQ
  • Storage: S3 compatível para documentos
  • Auth: Keycloak/Auth0/Clerk B2B
  • Observabilidade: OpenTelemetry + Grafana + Loki

Justificativa arquitetural

  • NestJS facilita modularização, DI e governança de domínio complexo.
  • PostgreSQL oferece integridade forte, JSONB para regras e boa capacidade analítica.
  • Redis/BullMQ desacopla jobs de cálculo, notificação e validação documental.
  • Next.js acelera construção de interfaces B2B densas com SSR quando necessário.
  • TypeScript reduz erro em regras financeiras/logísticas sensíveis.
9. Riscos e mitigação

O que pode falhar e como reduzir impacto

Riscos operacionais e técnicos

  • Dados físicos incorretos de peso/volume gerando rateio distorcido.
  • Conflitos comerciais por falta de explicabilidade no cálculo.
  • Fechamento com documentação incompleta e risco aduaneiro.
  • Excesso de customização por cliente tornando o motor inviável de manter.
  • Baixa liquidez inicial do marketplace em novas rotas.

Mitigações recomendadas

  • Validação dupla de peso/volume e alertas de outlier por SKU/histórico.
  • Memória de cálculo detalhada + versionamento + trilha de auditoria.
  • Workflow obrigatório de documentos com bloqueio por status.
  • Catálogo fechado de regras configuráveis antes de permitir scripts customizados.
  • Estratégia seed com operadores âncora e rotas de alta recorrência.
